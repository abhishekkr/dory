The jobs/services were being run in pre-baked containers over Kubernetes, Nomad, Mesos or others.
They were picking all configuration during spawn from environment variables injected. But some them required secrets to be shared with them which couldn't have been injected as environment variables as they are available to greater scope of maintainers and administrators with access to cluster, logs and others.
Secrets meant to be shared secretly, pulled from a source to a local configuration file which can be individually read by the spawned process. And cleaned from source and local store. If not accessed from secret store, it expunges it over time anyways.

Users call it with a authentication login name/identifier. If it pre-exists, you get HTTP-409 (Conflict).
If it can allocate, a token is generate and returned as response. If a timeout is not provided in request, default timeout of 5minute is configured. The storage of secret is in encrypted mode.
Now when this name/identifier is accessed with token, deciphered value is returned and store is expunged if explicitly not asked to retain.

Currently it'll enable local storage in AES encrypted blobs and HashiCorp's Vault as backend.
More can be added easily given project structure. Will soon provide external plug-in support for backends.
